# AGENTS.md — Meridian

Контекст проєкту для AI-агентів (Claude Code та ін.). Прочитай це перед роботою, щоб розуміти, над чим ідеться, не звіряючись щоразу з трекером.

---

## Що це

**Meridian** — персональний планувальник харчування для сім'ї. Бере плани від дієтолога (PDF), розкладає їх на окремі страви й перетворює на живий застосунок, який **сам збирає раціон на тиждень** із пулу страв, дотримуючись калорійності й структури дієтолога, але міксуючи страви з кількох базових планів.

Назва — від «меридіана» (полудень, поділ дня), що відображає ключову ідею **годинника прийомів їжі**.

### Три екрани
- **Сьогодні** — головна: круговий годинник дня, що підсвічує активний прийом їжі (сніданок / обід / вечеря / перекус) за поточним часом, + картка активної страви.
- **Тиждень** — згенерований план на 7 днів із денною калорійністю; кнопка «Перегенерувати».
- **Страви** — пул усіх страв, ручне додавання/редагування.

---

## Ціль

**Ціль продукту:** зняти з сім'ї ручну працю з планування харчування — щоб застосунок сам, щотижня, збирав різноманітний, але медично коректний раціон із перевірених дієтологом страв, і показував «що їсти зараз» без гортання PDF.

**Ціль поточного етапу (POC):** довести, що **генератор тижня працює** — на реальному пулі страв збирає валідний тиждень (правильні типи, калорійність у коридорі, без набридливих повторів, з міксом планів) і дає перегенерувати/замінити. Усе інше в POC обслуговує цю перевірку.

**Критерій успіху POC:** натиснув «Перегенерувати» — за секунди отримав інший валідний тиждень, який дотримується всіх правил дієтолога.

---

## Ядро продукту: генератор тижня

Це головна фіча. Для кожного дня добирає по одній страві на кожен слот так, щоб:

1. **Тип збігався** — сніданок ставиться лише на сніданок і т.д.
2. **Калорійність дня** трималась у коридорі від цілі дієтолога (за замовчуванням ±100 ккал).
3. **Антиповтор** — страва не повторюється частіше, ніж раз на N днів (N налаштовується).
4. **Мікс планів** — тиждень поєднує страви з різних базових планів, а не копіює один.

Технічно це **задача з обмеженнями** (constraint satisfaction). Реалізація для POC — жадібно-переборний підхід із випадковістю; якщо валідний набір не знайдено — послабити коридор і повідомити. **Важкий AI/ML не потрібен.**

Обов'язкові дії: «Перегенерувати тиждень» і «Замінити страву» (замінити один слот, не чіпаючи решту — як у Eat This Much).

## Модель даних «Страва»

Кожна страва — картка:

| Поле | Опис |
|------|------|
| `id` | Ідентифікатор (для посилань і антиповтору) |
| `name` | Назва (напр. «Гречка з куркою та овочами») |
| `type` | Тип слота: `breakfast` / `lunch` / `dinner` / `snack` |
| `calories` | Калорійність, ккал |
| `protein` / `fat` / `carbs` | БЖВ у грамах (необов'язкові) |
| `ingredients` | Список рядків |
| `portions` | Готові порції на людину з плану (MER-24): `{component, text}[]`, `text` — дослівно після «Готова порція:» (напр. «Ж - 130 г; Ч - 200 г»), `component` — складник страви або `null` |
| `source` | З якого плану дієтолога (напр. «План №2») — потрібно генератору для міксу |

---

## Модель даних «Профіль» (MER-21)

Підготовка до «Профіль на двох» (MER-17): плануємо не «для користувача», а для **активного профілю**.

| Поле | Опис |
|------|------|
| `id` | Ідентифікатор; профіль за замовчуванням має стабільний id `default` |
| `name` | Ім'я (напр. «Мій профіль») |
| `targetCalories` | Ціль калорійності дня, ккал |
| `corridor` | Ширина коридору, ±ккал |
| `goals` | Необов'язкові цілі БЖВ у грамах: `{protein, fat, carbs}` (`null` = не задано) |
| `mealIds` | Необов'язкова підмножина спільного пулу страв; `null` = увесь пул |
| `portion` | Порційна літера з плану дієтолога (MER-24): `"Ж"` / `"Ч"` / `null`. Картка «Сьогодні» і рецепт показують порцію активного профілю; без літери — всі порційні рядки дослівно |

Профілі й активний профіль живуть у `meridian.profiles.v1` (`ProfileStore`); вибір переживає перезавантаження. План тижня і календар ключуються профілем (`meridian.week.v1.<profileId>`, `meridian.calendar.v1.<profileId>`) — плани профілів не змішуються; пул страв спільний (`meridian.meals.v1`). Генератор приймає профіль (`options.profile` — об'єкт або id) і бере з нього ціль/коридор/пул; явні опції мають пріоритет. Допрофільні дані на старті автоматично мігрують під профіль за замовчуванням без втрат. UI перемикача профілів — MER-17.

## Стек (і чому саме такий)

Стек підпорядкований цілі POC — **найшвидше довести генератор, нічого не ускладнюючи.**

| Шар | Вибір | Чому (звʼязок із ціллю) |
|-----|-------|--------------------------|
| Застосунок | Один HTML-файл + вбудований JS (vanilla) | Логіка лишається в одному файлі, без фреймворка, бандлера й сервера застосунку. |
| Дані | `localStorage` браузера | Пул страв і згенеровані плани переживають перезавантаження. Не потрібні акаунти чи БД, щоб перевірити логіку. |
| UI | Tailwind CSS v4, мобільний-first, нижній таб-бар | Utility-класи збираються в статичний `tailwind.css`; у браузері Tailwind-runtime немає. |
| Генератор | Чистий JS, constraint satisfaction (жадібно + випадковість) | Задача проста; важкий AI лише сповільнив би POC. |
| Бекенд | **немає** (свідомо) | На етапі POC доводимо цінність, а не будуємо інфраструктуру. |

**Після POC (V1):** перенесення на React + невеликий бекенд для синхронізації між пристроями та імпорту PDF. Але **не раніше**, ніж генератор підтверджено.

Принцип: спершу довести цінність на тонкому прототипі, потім нарощувати.

### Збірка CSS

Runtime і деплой лишаються build-free: `index.html` та згенерований `tailwind.css`
зберігаються в репозиторії й віддаються як статичні файли. npm потрібен лише під час
розробки стилів. Node.js 20+ і npm потрібні лише авторам: після зміни
utility-класів або `tailwind.input.css` виконати `npm ci` (перший запуск) і
`npm run build:css`; для безперервної роботи є
`npm run watch:css`. Версії `tailwindcss` та `@tailwindcss/cli` зафіксовані на
`4.3.3`. Tailwind Preflight свідомо не підключено, щоб не змінювати нативний
вигляд form controls; потрібні reset-властивості задано utility-класами. Не
додавати Play CDN, PostCSS, JS-конфіг чи бандлер застосунку.

**Правило Tailwind-значень.** Не використовувати arbitrary values на кшталт
`py-[7px]`, `text-[13px]` чи `rounded-[10px]` і не відтворювати їх «точність»
вигаданими дробовими класами на кшталт `py-1.75`. Брати найближчий звичайний
токен зі стандартної шкали Tailwind (`py-2`, `text-sm`, `rounded-xl`). Arbitrary
variants на кшталт `[&.active]` або `[&_input]` дозволені: це селектори, а не
значення. Для значень, які не можна округлити без втрати поведінки (safe area,
динамічний колір профілю, точний breakpoint), додавати семантичний іменований
токен у `tailwind.input.css` і використовувати звичайний utility-клас. Після
заміни typography-токенів перевіряти браузером line-height, переповнення на
мобільній ширині та межі responsive-breakpoint.

## Ніша (чому це не «ще один» застосунок)

Найближчий аналог — **Eat This Much** (сам генерує тиждень під калорії, регенерує страву, робить список покупок). Але всі рішення на ринку (Eat This Much, PlateJoy, Mealime, Paprika) планують із **власних баз рецептів**. Відмінність Meridian: планування **із затвердженого дієтологом пулу** — медично безпечніше й персональніше.

## Принципи й конвенції

- **Мова інтерфейсу — українська.** Уся копірайтинг-частина, підписи, повідомлення — українською.
- **Жодних російських залежностей, сервісів чи джерел** — ні бібліотек, ні API, ні шрифтів російського походження. Це тверда вимога.
- **Не ускладнювати POC.** Ніякого бекенду, збірок, фреймворків на етапі POC — доки логіка генератора не підтверджена.
- **Класичні, перевірені рішення** над модними. Простий читабельний код.
- Одиниці калорій — ккал; БЖВ — грами.
- Слоти: `breakfast`, `lunch`, `dinner`, `snack`.

## Провенанс даних (тверде правило)

Кожне поле кожної страви й рецепта береться **лише** з одного з трьох джерел: (1) PDF-плану дієтолога (прикріплений до MER-8), (2) імпорту з цього PDF (MER-15), або (3) ручного вводу користувача, який переписує зі свого джерела.

Застосунок **ніколи не генерує, не вгадує й не доповнює** значень: назви, калорії, БЖВ, інгредієнти, кількості, кроки приготування, час, порції, фото. Немає в джерелі — поле лишається порожнім, а секція UI ховається. Жодних «приблизних», «типових» чи «схожих» значень.

Що фактично дає поточний PDF (`Menu_2_tyzhni`, засіяно як `DEMO_MEALS`): назву, тип прийому, калорійність, інгредієнти (часто без точних грамовок), номер тижня, готові порції на людину — рядки «Готова порція: Ж - …; Ч - …» (MER-24, зберігаються дослівно); для частини страв — БЖВ. Кроків приготування й часу в ньому немає — тож ці поля порожні за замовчуванням, доки не з'являться в джерелі. Калорійності на порцію PDF теж не дає — її немає, доки не впише користувач чи не дасть джерело. Демо-пул має **точно** відповідати PDF; страва без реального сорсу — помилка.

**Виняток для калорій (свідомий, для POC).** Поточний PDF дає лише одну цифру (десерт ≈185 ккал), а на калоріях тримається коридор генератора (MER-10). Тому калорійність решти страв демо-пулу — **приблизні оцінки із загальних таблиць харчування** (порція «Ч»). Це дозволено **лише** з явним маркуванням: у даних `caloriesApprox: true`, в UI — префікс «≈». Реальне (не-approx) значення має тільки те, що прямо в PDF (банановий хліб 185 ккал). Замінити на цифри дієтолога, щойно з'являться (MER-26). Виняток стосується **лише калорій** — назви, інгредієнти, БЖВ, кроки тощо лишаються строгими.
